# Refactoring — 안전한 코드 리팩토링 가이드

기존 에이전트들(frontend-dev, backend-dev, ai-engineer)이 리팩토링 작업 시 공통으로 참조하는 워크플로우와 안전 규칙.

## 대상 에이전트

모든 에이전트가 공유한다:
- `frontend-dev` — React 컴포넌트, 훅, 스토어 리팩토링
- `backend-dev` — Supabase 스키마, API, 쿼리 리팩토링
- `ai-engineer` — 프롬프트 구조, 분석 파이프라인 리팩토링
- `qa-engineer` — 리팩토링 후 리그레션 검증

## 핵심 원칙

1. **동작 보존**: 리팩토링은 외부 동작을 바꾸지 않고 내부 구조만 개선한다
2. **점진적 변경**: 한 번에 하나의 리팩토링만 수행한다. 여러 리팩토링을 묶지 않는다
3. **검증 후 진행**: 각 단계마다 빌드/린트 통과를 확인한 후 다음으로 넘어간다
4. **되돌림 가능**: 커밋 단위로 분리하여 문제 시 개별 롤백이 가능하게 한다

## 리팩토링 워크플로우

```
1. 분석  →  2. 계획  →  3. 실행  →  4. 검증  →  5. 정리
```

### 1단계: 분석 — 무엇을 왜 리팩토링하는가

- 대상 코드를 읽고 현재 구조의 문제점을 파악한다
- 리팩토링 동기를 명확히 한다:

| 동기 | 신호 | 예시 |
|------|------|------|
| **크기 초과** | 파일 200줄+, 함수 50줄+ | InterviewPage.jsx가 500줄 |
| **중복 코드** | 동일 로직 2곳 이상 | 여러 페이지에서 같은 API 호출 패턴 |
| **혼합 책임** | 한 파일이 여러 역할 | 컴포넌트에 데이터 로직 + UI + 비즈니스 로직 |
| **깊은 결합** | 변경 시 연쇄 수정 필요 | 스토어 변경 시 5개 컴포넌트 수정 |
| **성능 문제** | 불필요한 리렌더링, 느린 로딩 | 전체 리스트 리렌더링 |

### 2단계: 계획 — 어떻게 리팩토링하는가

영향받는 파일 목록을 작성하고, 리팩토링 순서를 결정한다. `_workspace/refactor_plan.md`에 계획을 기록한다:

```markdown
## 리팩토링 계획
- **대상**: [파일/모듈]
- **동기**: [문제점]
- **방법**: [적용할 패턴]
- **영향 파일**: [변경되는 파일 목록]
- **위험도**: 낮음/중간/높음
- **검증 방법**: [빌드, 수동 테스트, 특정 플로우 확인]
```

### 3단계: 실행 — 패턴별 리팩토링

아래 패턴 카탈로그에서 적합한 패턴을 선택하여 적용한다.

### 4단계: 검증

매 리팩토링 단계 후 반드시 확인한다:
- `npm run build` — 빌드 통과
- `npm run lint` — ESLint 통과
- 영향받는 기능의 수동 테스트 시나리오 (계획서에 명시)

### 5단계: 정리

- 미사용 import 제거
- 빈 파일/디렉토리 삭제
- 변경 사항을 커밋 메시지에 명확히 기록

## 패턴 카탈로그

### React 컴포넌트 리팩토링

#### 컴포넌트 분리 (Extract Component)
**신호**: 컴포넌트 200줄 초과, JSX 내 반복 블록
**방법**:
1. 분리할 UI 블록 식별
2. 필요한 props 정의
3. 새 파일로 추출
4. 원본에서 import하여 교체

**주의**: 상태(state)를 함께 추출할지, props로 내릴지 판단한다. 상태가 부모에만 필요하면 props drill, 여러 곳에서 필요하면 Zustand로 이동한다.

#### 커스텀 훅 추출 (Extract Hook)
**신호**: 컴포넌트 내 상태+이펙트 로직이 UI와 섞여 있음
**방법**:
1. 관련 useState, useEffect, 핸들러 함수를 묶어 식별
2. `src/hooks/use{Name}.js`로 추출
3. 반환값은 컴포넌트가 필요로 하는 것만

**이 프로젝트의 기존 훅 패턴 참조**: `useMediaStream`, `useMediaRecorder`, `useFrameCapture`, `useAudioLevel`, `useAnalysis`

#### Zustand 스토어 분리 (Split Store)
**신호**: 스토어가 비관련 상태를 함께 관리, 스토어 파일 150줄+
**방법**:
1. 관련 상태끼리 그룹핑
2. 독립적인 스토어로 분리
3. 스토어 간 의존이 필요하면 `getState()` 사용

**이 프로젝트 스토어**: `authStore`, `interviewStore`, `settingsStore`

### 백엔드 리팩토링

#### API 함수 분리 (Extract Service)
**신호**: 서버리스 함수에 비즈니스 로직이 인라인으로 작성됨
**방법**:
1. 요청 파싱 / 비즈니스 로직 / 응답 포맷팅을 분리
2. 공통 로직은 `src/lib/`에 유틸리티로 추출

#### Supabase 쿼리 최적화
**신호**: N+1 쿼리, 불필요한 전체 컬럼 SELECT
**방법**:
1. `select('*')` → 필요한 컬럼만 명시
2. 관련 테이블은 join으로 한 번에 조회
3. 자주 사용하는 쿼리는 RPC 함수로 전환

### AI 파이프라인 리팩토링

#### 프롬프트 모듈화 (Extract Prompt Template)
**신호**: 프롬프트 문자열이 API 핸들러에 인라인으로 작성됨
**방법**:
1. 프롬프트를 별도 모듈/상수로 추출
2. 변수 부분을 템플릿 파라미터로 분리
3. 프롬프트 버전 관리 가능하게 구조화

## 위험도별 리팩토링 규모

| 위험도 | 범위 | 예시 | 검증 수준 |
|--------|------|------|----------|
| **낮음** | 단일 파일 내부 | 변수 리네이밍, 함수 추출 | 빌드 통과 |
| **중간** | 2~5개 파일 | 컴포넌트 분리, 훅 추출 | 빌드 + 해당 페이지 동작 확인 |
| **높음** | 5개+ 파일 또는 데이터 흐름 변경 | 스토어 재구조화, API 변경 | 빌드 + 전체 면접 플로우 확인 |

## 금지 사항

- 리팩토링과 기능 변경을 같은 커밋에 섞지 않는다
- 동작하는 코드를 "더 우아하게" 만들기 위해 리팩토링하지 않는다 — 구체적 문제가 있을 때만
- 리팩토링 중 발견한 버그는 별도 이슈로 분리한다 — 리팩토링 커밋에 버그 수정을 끼워넣지 않는다
